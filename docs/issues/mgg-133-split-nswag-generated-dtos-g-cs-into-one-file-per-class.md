# Split NSwag-generated Dtos.g.cs into one file per class

## Problem

NSwag generates all DTOs into a single `src/Presentation/API/Generated/Dtos.g.cs`. As the API grows, this file becomes large and creates two issues:

1. **Agent context waste** — AI agents read the full file to understand a single DTO, burning context window on hundreds of unrelated types
2. **Discoverability** — harder to grep/navigate than one-file-per-class

## Proposal

Add a post-generation MSBuild target that splits `Dtos.g.cs` into individual files using Roslyn's `CSharpSyntaxTree`, wired as an `AfterTargets` of the existing NSwag generation step.

### Output structure

```
src/Presentation/API/Generated/
  AccountResponse.g.cs
  CreateAccountRequest.g.cs
  UpdateAccountRequest.g.cs
  UserRolesResponse.g.cs
  ...
```

Each split file includes a header comment with the spec hash:

```csharp
// <auto-generated>
// Generated from openapi/spec.yaml (SHA-256: a1b2c3d4...)
// Delete this file if the spec changes and DTOs are regenerated.
// </auto-generated>
```

The monolithic `Dtos.g.cs` is deleted after splitting.

### Conditional regeneration (spec hash in generated files)

The splitter must NOT re-run on every build. The spec hash is embedded in every generated file header, so no separate hash file is needed.

Build-time logic:

1. Compute SHA-256 of `openapi/spec.yaml`
2. Read the hash from any existing `Generated/*.g.cs` file header (they all share the same hash)
3. **Match → skip** both NSwag generation and splitting entirely
4. **Mismatch (or no files exist) → regenerate**: run NSwag, split, write files with new hash
5. First build (empty `Generated/`) always regenerates

This should be implemented as an MSBuild `<Target>` with a `Condition` check so `dotnet build` short-circuits cleanly.

### Stale DTO cleanup

When the spec hash changes, ALL existing `*.g.cs` files in `Generated/` are deleted before writing the new split files. This is safe because:

* Regeneration is atomic: NSwag produces the full output, then we split it
* It only happens when the spec actually changed (hash mismatch)
* Any file from a previous spec version is by definition stale

This naturally handles DTO removals: if a schema is deleted from the spec, its `.g.cs` file is wiped in the clean step and never re-created.

### Implementation notes

* Use `Microsoft.CodeAnalysis.CSharp` (Roslyn) to parse the generated file and extract top-level type declarations
* Each type gets its own file named `{TypeName}.g.cs` with the original file-scoped namespace, required usings, and the spec-hash header comment
* The splitter can be a small console tool in `tools/` or a dotnet-script, invoked via `<Exec>` in the API `.csproj`
* Keep it simple: no need for incremental per-type diffing — the spec-hash gate is sufficient
* The split files should be **checked in** (not gitignored) so IDEs and agents can discover individual DTOs without building first

## Acceptance criteria

- [ ] `dotnet build` produces individual `.g.cs` files in `Generated/`, one per DTO class
- [ ] Each file has a header comment containing the SHA-256 of `spec.yaml`
- [ ] Consecutive `dotnet build` with no spec changes does NOT re-run generation or splitting
- [ ] Modifying `spec.yaml` and rebuilding regenerates, deletes old files, and writes new split files
- [ ] Removing a schema from `spec.yaml` and rebuilding removes the corresponding `.g.cs` file
- [ ] `dotnet format --verify-no-changes` passes (generated files excluded or formatted)
- [ ] Drift check still works (operates on NSwag's JSON export, not the split C# files)
- [ ] All existing tests pass

## Open questions

* Should the splitter be a standalone console tool (`tools/DtoSplitter/`) or a `dotnet-script` file? Console tool is more debuggable; script is lighter.
